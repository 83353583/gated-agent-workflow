# Gated Agent Workflow — Playbook

> 工具无关的核心流程。Agent 应优先遵守本文，再参考各工具 adapters。

## 0. 硬约束（任何步骤都适用）

1. **无验收不得宣称完成。** 验收优先为可执行命令；没有命令则写清人工检查清单。
2. **一步只做一步。** 禁止提前实现后续步骤；禁止「顺手重构」范围外模块。
3. **每步结束必须硬停。** 不要在同一长会话中连续推进多步（防止自动压缩上下文丢失约束）。
4. **每步验收通过后，必须产出下一份可复制的 Handoff Prompt**（见 `templates/handoff.md`）。
5. **事实与计划分离。** 「已完成」只写已落地且可验证的内容；猜测写入「已知坑/待确认」。
6. **维护期默认代码为真相。** 文档与聊天冲突时，以当前代码 + 你刚跑通的验收为准，并回写文档。
7. **模型档位按任务类型选择**（见 §4），不要默认全程高强模型。

## 2. 入口 A：新项目启动（Bootstrap）

### 2.1 流程骨架

```text
Discover（需求讨论，可同会话多轮）
  → 人类确认「范围可冻结」
  → 冻结 SCOPE（MVP / 一期 / 二期）
  → 写出约束（技术栈可暂空）
  → 技术选型一页纸（可延后到「会影响目录/依赖」的第一步之前）
  → 步骤表（每步：目标 / 默认模型档 / 升级条件 / 验收命令 / 是否触发选型冻结）
  → 对每一实现步：
       新开 Agent + 粘贴 Handoff
       → 只实现本步
       → 跑验收命令并回报输出
       → 人类验收通过
       → 产出下一份 Handoff
       → 硬停 →（按需切模型）→ 下一步
```

### 2.2 阶段说明

#### 阶段 0 — Discover（需求讨论）

- **模板：** `templates/discovery.md` → 写入项目 `docs/ai-workflow/DISCOVERY.md`。
- **目标：** 对齐问题、用户、场景、成功标准、非目标、待澄清问题、MVP 初判。
- **模型：** 中高。拆歪需求比写错 CRUD 更贵。
- **会话策略：** 允许同会话多轮问答；**不必**每轮硬停。  
  硬切点是人类确认「范围可冻结」之后：再开（或同会话切换到）SCOPE / 文档阶段，不要把长讨论史当作实现上下文。
- **产出：** 更新后的 `DISCOVERY.md`；冻结后产出 SCOPE 草稿，或「起草 SCOPE」的可复制 Handoff。
- **禁止：** 写业务代码；写实现步骤表并开工；在未要求时强行冻结技术栈。
- **门禁：** 未确认「范围可冻结」前，不得进入阶段 1 之后的实现准备（可先 scaffold 空模板）。
- **跳过条件（谨慎）：** 用户已自带清晰 SCOPE/PRD 且声明跳过讨论 → 可直接阶段 1，但仍建议把原文链到 `DISCOVERY.md` 或 SCOPE 背景里留一行。

#### 阶段 1 — 范围冻结（SCOPE）

- 使用 `templates/scope.md`；内容应能从 `DISCOVERY.md` 直接提炼，不要另起一套互相矛盾的故事。
- **Must（MVP）**：没有就不能演示的最短路径。
- **Should（一期）**：明确依赖 MVP 的什么。
- **Could（二期）**：只写边界一句话，不写细设计。
- **门禁：** 人类确认 SCOPE 后才能写步骤表 / 开始实现步。

#### 阶段 2 — 技术选型（可延后）

- 使用 `templates/architecture.md`。
- 只写：**结论、理由、约束、禁止事项**；不写长文对比。
- **冻结点：** 第一个会影响目录结构或依赖安装的步骤之前，必须完成并写入全局真相。
- 冻结后默认只读；重选必须人类明确授权，并记入决策日志。

#### 阶段 3 — 步骤表

- 使用 `templates/step-table.md`。
- 优先 **垂直切片**（一条用户路径打通），避免「先写完所有后端」。
- 每步必须有：交付物、相关路径提示、验收命令、模型档、升级条件。
- **门禁：** 人类确认步骤表后，才能开始 Step 1。

#### 阶段 4 — 分步实现

- 每步新会话；上下文 = 全局真相摘要 + 本步 Handoff（不要贴整段聊天史）。
- Handoff 目标长度：**约 800–1500 汉字/词**（一到两屏）。更长 → 步骤太大，应拆步。
- 结束后自检 Handoff（见 §5.2），再交给人类。

#### 阶段 5 — MVP 完成

- 更新 SCOPE「已交付」。
- 补决策日志。
- 归档最终步骤表与关键 Handoff（可选）。

### 2.3 详细设计何时写

| 文档 | 时机 | 深度 |
|------|------|------|
| DISCOVERY | 从零最先 | 一页；讨论中迭代 |
| SCOPE / 步骤表 / 选型结论 | 冻结范围后、开干前 | 完整但短 |
| 接口 / 库表 / 模块细设计 | **做到该步再写** | 按需 |

## 3. 入口 B：维护 / 新需求（Change Request）

不要复用完整 Bootstrap。单位是 **变更**，不是整项目。

### 3.1 流程骨架

```text
填写变更单（目标 + 验收 + 非目标）
  → 只读勘察 Agent（默认不改代码）
  → 人类确认影响面与步骤拆分
  → 分步实施（每步 Handoff + 硬停 + 回归验收）
  → 文档同步 Agent（只改 SCOPE/ARCHITECTURE/决策日志，不改业务代码）
```

### 3.2 只读勘察（强制，除非走小改旁路）

对 Agent 的固定口令：

> 按变更单做只读勘察，不改代码。输出：影响文件、现有行为（据代码）、风险与回归面、步骤拆分、模型档建议、实施 Handoff。

勘察产出必须能回答：下一实施 Agent **不读旧聊天**也能开工。

### 3.3 实施与回归

- Handoff 必须额外包含：**回归范围**、**禁止改动的模块**。
- 验收 = 本步新标准 + 回归命令/路径。
- 禁止范围外重构。

### 3.4 文档回写

变更验收通过后，单独开一个低/中档任务：

> 本变更已通过，只更新 SCOPE / ARCHITECTURE / 决策日志，不改业务代码。

### 3.5 小改旁路

同时满足可走旁路：

- 影响面可预判为极少文件（通常 ≤3）
- 无架构/选型变更
- 有清晰复现或验收方式

旁路：`变更单（极短）→ 单步实施 → 验收 → 可选一行决策日志`。  
不确定时，仍走完整勘察。

### 3.6 何时在维护期再做 Discover

变更单里「目标/验收」写不清，或可能改动一期/二期边界时：先用半页 `DISCOVERY.md`（或变更单内嵌同样字段）对齐，再勘察/实施。日常小改不强制。

### 3.7 Bug 修复包

```markdown
## Bug
- 复现步骤：
- 期望 / 实际：
- 日志/截图：（如有）

## 约束
- 先定位再最小修复
- 禁止大范围重构
- 验收必须覆盖原复现步骤（反向验证）
```

## 4. 模型档位策略

| 档位 | 适用 |
|------|------|
| **低** | 样板、文案、格式化、按已定接口做 CRUD、文档同步 |
| **中** | 常规功能、测试补齐、局部重构（仍在步内）、勘察 |
| **高** | 需求拆解、架构/鉴权/数据模型、疑难排障、跨模块设计 |

**默认用中档。** 步骤表写「默认档 + 升级条件」，例如：

- 默认：中  
- 升级到高：若涉及鉴权模型 / 数据迁移 / 多模块协议变更  

人类可在门禁处按档位切换模型后再开下一 Agent。

## 5. Handoff 规范

### 5.1 必需章节

按 `templates/handoff.md`，至少包含：

1. 本步目标 / 明确不做  
2. 全局约束摘要（栈、范围、禁止项）  
3. 上一步已完成（事实）  
4. 本步输入（优先读的文件路径）  
5. 交付物  
6. 验收命令与通过标准  
7. 模型档建议  
8. 结束时必须输出：变更清单 + 验收结果 + **下一份 Handoff**

维护步进再加：回归范围、禁止改动。

### 5.2 交出前自检

- [ ] 下一 Agent 不读旧聊天是否仍够用？  
- [ ] 「已完成」是否混入未验证猜测？  
- [ ] 是否超过约两屏？该不该拆步？  
- [ ] 验收命令是否可直接复制执行？  

---

## 6. 给人类的操作清单

每步你只做四件事：

1. 粘贴 Handoff，开新 Agent（按档位选模型）  
2. 看它是否越权、是否跑了验收  
3. 点头或打回  
4. 复制它产出的下一 Handoff，进入下一步  

不要依赖「接着上一段聊天说：继续」。

## 9. 从零开项目：可简化到什么程度

全局 Skill 已装好时，人类侧可收成下面几句（细节由 Agent 按模板落盘）：

```text
1. （可选）建空目录 / 开仓库
2. 「初始化门禁工作流」→ 确认 scaffold
3. 「我要做 ___，先需求讨论」→ 多轮对齐 → 你说「范围可冻结」
4. 确认 SCOPE + 步骤表（选型可仍空）
5. 按 Handoff 一步步实现（每步新会话）
```

不必每次手写长 Prompt；也不必在开干前写完所有详细设计。  
**不能再删的：** Discover 冻结门禁、SCOPE、分步验收 + Handoff 硬停。
